# vite-swap 项目功能与技术实现说明

本文档用于整理当前项目的技术栈、目录结构、已实现功能、核心接口、关键方法和业务流程，方便后续维护、交接和二次开发。

## 1. 项目概述

`vite-swap` 是一个面向移动端 / 小程序 WebView 场景的 DW20 与 FTC 兑换页面项目。

当前项目主要实现：

- 首页资产兑换页面
- 猜奖币兑换：DW20 兑换 FTC
- 奖金存入钱包：FTC 存入无链钱包入口
- 兑换前手续费预估
- 二次确认弹框
- 交易提交中弹框
- 无链钱包授权与余额获取
- 钱包地址查询与缓存
- 首页最近 5 条兑换记录
- 独立兑换记录页面
- 全部记录 / 猜奖币兑换 / 奖金存入钱包 三个 tab
- 记录分页与上拉加载更多
- 记录状态中文展示与颜色区分
- vConsole 调试面板，生产环境也会加载
- 本地开发和生产部署统一走同源代理路径，避免小程序 WebView 跨域和 HTTP 请求拦截问题

## 5. 路由实现

路由文件：`src/App.tsx`

项目没有引入 `react-router`，而是使用浏览器原生 History API 实现了轻量路由。

当前页面：

| 路径 | 页面组件 | 说明 |
| --- | --- | --- |
| `/` | `HomePage` | 首页兑换页面 |
| `/records` | `RecordsPage` | 兑换记录页面 |

核心方法：

- `getPageFromLocation()`：根据 `window.location.pathname` 判断当前页面。
- `navigateTo(nextPage)`：使用 `window.history.pushState` 修改地址栏并切换页面状态。
- `popstate` 监听：支持浏览器返回时同步页面状态。

## 10. 钱包地址获取与缓存

文件：`src/lib/wallet.ts`

接口：

```text
GET /wallet-api/pub/wallet/wulian?userId={accountId}
```

真实目标：

```text
http://mmt-user.budingcc.cc/pub/wallet/wulian?userId={accountId}
```

成功返回示例：

```json
{
  "walletAddress": "0x61e026f9ad0af11c2900ada0d59b9dd32f023e98",
  "status": "success"
}
```

未注册返回示例：

```json
{
  "walletAddress": "",
  "status": "NoRegistered"
}
```

实现方法：

- `fetchWalletInfo(accountId)`：根据无链授权账号查询 EVM 钱包地址。
- `getStoredWalletAccountId()`：从 `localStorage` 读取上次成功查询的钱包账号。
- `getStoredWalletAddress()`：从 `localStorage` 读取上次成功查询的钱包地址。
- `walletRequestMap`：使用 `Map<string, Promise<WalletResponse>>` 复用同一个账号的进行中请求，避免重复请求。

缓存策略：

- `status === "success"` 且 `walletAddress` 不为空时：
  - 写入 `localStorage.wulian_wallet_account_id`
  - 写入 `localStorage.wulian_wallet_address`
- `status === "NoRegistered"` 或钱包地址为空时：
  - 清空本地钱包缓存
  - 页面使用 antd `message.warning("用户未注册，请先注册钱包")`

## 11. 无链授权与账户资产

首页文件：`src/pages/HomePage.tsx`

项目使用 `mini-program-api` 的 `BrowserProvider`：

```ts
new miniProgramApi.BrowserProvider(window.noChain)
```

### 11.1 授权登录信息 sessionStorage 缓存

当前首页将授权返回信息存入 `sessionStorage`：

```text
wulian_auth_user_info
```

相关方法：

- `getStoredAuthInfo()`：从 `sessionStorage` 读取授权信息。
- `getStoredAuthAccountId()`：从授权信息里取 `data.accountId`。
- `setStoredAuthInfo(authInfo)`：写入或清除授权信息。

流程：

1. 页面初始化时等待 `window.noChain` 注入。
2. 创建 `BrowserProvider`。
3. 优先读取 `sessionStorage.wulian_auth_user_info`。
4. 如果有 `accountId`，直接使用，不再重复调用授权。
5. 如果没有缓存，调用 `requestAuth`：

```ts
noChainProvider.current.requestAuth({
  type: "auth_account",
  scope: "userInfo",
  actions: [],
})
```

6. 授权成功后，将返回信息写入 `sessionStorage`。
7. 使用 `accountId` 查询钱包地址。
8. 调用 `getAccount()` 获取 DW20 余额。

### 11.2 DW20 余额获取

调用：

```ts
noChainProvider.current.getAccount()
```

返回中的 `balance` 示例：

```json
[
  {
    "coin": "DW20",
    "available_balance": "100000.00000000",
    "total_balance": "100000.00000000"
  }
]
```

首页只取：

```ts
balance.find((item) => item.coin === "DW20")?.available_balance
```

用途：

- 显示首页“可用 DW20”。
- 限制输入框最大可输入值。
- 兑换成功后重新调用 `getAccount()` 刷新最新 DW20 余额。

相关方法：

- 初始化流程里直接调用 `getAccount()`。
- `refreshDw20Balance()`：交易成功后刷新余额。

### 12.1 输入框规则

输入框只允许整数：

```ts
const integerAmount = nextAmount.replace(/\D/g, "").replace(/^0+(?=\d)/, "")
```

限制逻辑：

- 非数字直接忽略。
- 空值则清空输入。
- 最大值不能超过当前 DW20 可用余额。
- 猜奖币兑换模式下，最大值来自 `dw20AvailableBalance`。
- 奖金存入钱包模式下，最大值来自 `availableAmount`。

### 12.2 金额大数转换

首页内部有两种金额单位转换。

#### 12.2.1 交易提交金额

方法：

```ts
parseNearAmount(amountText)
```

说明：

- 按 24 位小数转换。
- 用于 `sendContractTxRaw` 的 `actions.amount`。
- 例如用户输入 `13`，会转成链上最小单位字符串。

#### 12.2.2 记录列表金额

方法：

```ts
formatTokenAmount(amount)
```

说明：

- 按 18 位小数转换。
- 用于后端记录接口返回的 `amount`。
- 首页最近记录和记录页都按这个逻辑展示。

### 12.3 手续费预估

方法：

```ts
estimateTransferFee({ accountId, amount, receiverId })
```

请求地址：

```text
POST /rpc
```

请求体：

```json
{
  "jsonrpc": "2.0",
  "id": "dontcare",
  "method": "query",
  "params": {
    "request_type": "transfer_fee",
    "finality": "final",
    "account_id": "1002159596.user",
    "receiver_id": "dw20-lock.contract",
    "symbol": "TDW20",
    "amount": "用户输入数量转换后的 24 位大数字符串",
    "fee_symbol": ["TDW20"]
  }
}
```

兼容的响应结构：

```ts
response.data.result?.fees?.[0]
response.data.result?.result?.fees?.[0]
```

处理方式：

- 将 `fees[0]` 转为字符串。
- 调用 `formatNearAmount` 转成人类可读数量。
- 使用 `Math.ceil` 向上取整。
- 请求失败时返回 `0`。

展示位置：

- 点击“立即兑换”后，先预估手续费。
- 手续费预估完成后，打开二次确认弹框。
- 弹框显示：
  - 兑换数量
  - 手续费
  - 共计支付

### 12.6 猜奖币兑换提交流程

核心方法：

```ts
submitExchange()
sendContractMethod()
```

完整流程：

1. 用户点击“立即兑换”。
2. 调用 `estimateTransferFee` 预估手续费。
3. 打开二次确认弹框。
4. 用户点击确定。
5. 设置 `isSubmittingExchange = true`。
6. 调用 `sendContractMethod()`。
7. 校验无链插件是否初始化。
8. 校验钱包地址是否存在。
9. 校验钱包账号是否存在。
10. 组装合约调用参数 `opt`。
11. 调用 `sendContractTxRaw` 获取 `txRaw`。
12. 调用 `sendBroadcastTx` 广播交易。
13. 调用 `sendBroadcastHash` 确认广播 hash。
14. 调用后端接口轮询充值状态。
15. 成功后刷新 DW20 余额。
16. 成功后清空输入框。
17. 不管成功或失败，都刷新首页最近 5 条兑换记录。
18. 关闭提交中状态。

### 12.7 合约调用参数

当前参数：

```ts
const opt = {
  receiverId: "dw20-lock.contract",
  sender_account_id: walletAccountId,
  actions: {
    method_name: "deposit",
    args: {
      evm_address: walletAddress,
    },
    max_gas: 300000000000000,
    amount: parseNearAmount(paymentAmount),
    symbol: "TDW20",
    fee_symbol: "TDW20",
  },
}
```

注意：

- 当前运行时要求 `actions` 是对象。
- SDK 类型声明可能与运行时不一致，所以代码中通过类型适配保留对象结构。
- `amount` 使用用户输入值转换后的 24 位大数字符串。

### 12.8 后端充值状态轮询

方法：

```ts
pollDepositStatus(chainlessTxHash)
```

请求：

```text
GET /wallet-api/pub/bridge/deposit?chainless_tx_hash={hash}
```

真实目标：

```text
http://mmt-user.budingcc.cc/pub/bridge/deposit?chainless_tx_hash={hash}
```

状态：

| status | 说明 | 处理 |
| --- | --- | --- |
| `success` | 充值成功 | 返回 `true` |
| `error` | 充值失败 | 返回 `false` |
| `pending` | 充值处理中 | 继续轮询 |

轮询规则：

- 每 `0.7s` 查询一次。
- 最多查询 `40` 次。
- 40 次后仍未成功，则视为失败。

### 13.1 三个 tab 对应接口

| tab | recordFilter | 接口 | 类型 |
| --- | --- | --- | --- |

实际请求示例：

```text
GET /bridge-api/pub/bridge/transactions?evm_address={walletAddress}&page=1&page_size=20
GET /bridge-api/pub/bridge/deposits?evm_address={walletAddress}&page=1&page_size=20
GET /bridge-api/pub/bridge/withdrawals?evm_address={walletAddress}&page=1&page_size=20
```

### 13.2 钱包地址来源

记录页不会主动调用无链授权。

它通过：

```ts
getStoredWalletAccountId()
fetchWalletInfo(accountId)
```

从本地缓存的钱包账号重新查询钱包地址。

如果没有账号或没有钱包地址：

- 不请求记录接口。
- 列表置空。
- 展示空数据。

### 13.3 请求去重

记录页使用：

```ts
const inFlightRequests = new Map<string, Promise<ListResponse<...>>>()
```

作用：

- 同一个 URL 的请求如果正在进行，直接复用该 Promise。
- 避免 React StrictMode、tab 切换、滚动触底等场景造成重复请求。

### 13.4 请求竞态处理

记录页使用：

```ts
requestKeyRef.current = requestKey
```

每次请求生成 key：

```ts
`${walletAddress}-${recordFilter}-${nextPage}-${mode}`
```

响应回来后检查：

```ts
if (requestKeyRef.current !== requestKey) return
```

作用：

## 17. 当前接口汇总

### 17.1 钱包地址接口


用途：

### 17.4 全部记录接口

用途：

### 17.5 猜奖币兑换记录接口

### 17.6 奖金存入钱包记录接口

### 18.1 首页初始化流程


## 19. 当前实现注意事项

1. 首页“猜奖币兑换”已接真实提交流程，“奖金存入钱包”当前只打印日志。
2. 授权信息使用 `sessionStorage`，同一个小程序会话内不会重复请求授权；关闭会话后需要重新授权。
3. 钱包地址使用 `localStorage` 缓存，用于首页和记录页复用。
4. 所有接口都应使用 `/wallet-api`、`/bridge-api`、`/rpc` 代理路径，不建议在页面里直接写完整远程地址。
5. 小程序 WebView 中出现 `HTTP Status 0` 通常与跨域、HTTP 明文请求、证书、WebView 安全策略有关，因此项目统一走同源代理。
6. `sendContractTxRaw` 当前运行时要求 `actions` 是对象，不能改成数组。
7. 合约提交金额使用 24 位小数转换；记录列表金额使用 18 位小数格式化，两者不要混用。
8. React StrictMode 可能导致开发环境 effect 执行两次，当前记录请求已通过请求 Map 和 requestKey 做了去重与竞态保护。
9. vConsole 在生产环境也会显示，方便小程序真机调试；上线正式环境如不希望展示，可后续增加环境开关。
10. `status` 如果接口缺失，部分归一化逻辑会默认 `success`，如果后端一定返回状态，建议长期保持接口字段完整。

## 20. 后续可优化方向

- 将首页中的无链授权、余额刷新、金额转换、记录格式化抽成独立 hook 或工具函数。
- 将首页和记录页重复的 `formatTokenAmount`、`formatStatus`、`getStatusClassName`、`formatTimestamp` 合并到公共工具文件。
- 为奖金存入钱包补充真实提交流程。
- 为交易失败原因 `failed_reason` 增加展示。
- 为 RPC 预估手续费失败增加更明确 UI 状态，例如“手续费预估失败，请重试”。
- 增加接口错误码统一处理。
- 增加生产环境 vConsole 开关。
- 如果路由继续增加，可以引入 `react-router` 或封装当前 History API 路由。
